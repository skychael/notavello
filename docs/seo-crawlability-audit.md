# Notavello technical SEO crawlability audit

Audit date: 2026-07-20

Scope: repository configuration, generated static HTML, internal discovery, sitemap and robots controls, Cloudflare Pages Worker behavior, GitHub Actions publication flow, and sampled live responses from `https://notavello.com`.

## Executive summary

The 51 Search Console URLs reported as **Discovered - currently not indexed** are not explained by a site-wide robots block, `noindex`, client-only article rendering, or a universal bot challenge. The repository serves blog articles as static HTML, article canonicals are self-referencing, `robots.txt` allows the site, the blog archive contains ordinary links to every current article, and sampled live requests using a Googlebot user agent returned HTTP 200 with the article HTML in the response.

There is, however, a confirmed discovery defect: automated blog commits do not reliably refresh `sitemap.xml`. The current sitemap is two articles behind the repository. The blog workflow pushes with GitHub Actions credentials, and that push does not trigger the separate sitemap workflow. The sitemap generator also includes files based only on their filename/location; it does not check canonical, robots, duplicate, or indexability signals, and it does not publish meaningful `lastmod` values.

The strongest remaining explanation for the large Search Console cohort is crawl prioritization and/or Google's page-quality selection rather than an outright crawl prohibition. That cannot be confirmed from repository code alone and should be evaluated after the technical fixes are deployed and recrawled. The separate content-quality report generated by the audit tool must be reviewed editorially; article copy must not be rewritten automatically.

## Architecture and deployment

- Framework: no application framework; static HTML/CSS/JavaScript files.
- Hosting: Cloudflare Pages, inferred from `_redirects`, the Pages asset binding used by `_worker.js`, and live Cloudflare response headers.
- Routing: directory `index.html` files map to trailing-slash URLs. Cloudflare Pages applies `_redirects` for selected legacy paths.
- Server rendering: `_worker.js` fetches the requested static HTML from `env.ASSETS`, replaces `<!--HEADER-->` and `<!--FOOTER-->` with static partials, and returns complete HTML. It does not generate article bodies client-side.
- Deployment: pushes to `main` are the apparent Cloudflare Pages deployment source. GitHub Actions generates daily posts, refreshes blog pagination/archive files, commits them to `main`, and separately generates/commits `sitemap.xml` on pushes to `main`.

## SEO source map

- `robots.txt`: committed root static file.
- XML sitemap: root `sitemap.xml`, generated by `.github/scripts/generate-sitemap.js` and `.github/workflows/generate-sitemap.yml`.
- Canonicals and meta robots: hard-coded in each HTML document and in the blog post template/generator output.
- Blog index/archive: `pages/blog/index.html`, `pages/blog/page-*/index.html`, and `pages/blog/all-posts/index.html`; maintained by `blog-automation/split-blog-index.js` during post generation.
- Shared navigation: `_header.html` and `_footer.html`, injected in server responses by `_worker.js`.
- Homepage discovery: `index.html` links directly to `/pages/blog/`; the shared header/footer currently do not contain a Blog link.

## Confirmed problems

### 1. Automated posts can be absent from the sitemap

Affected URLs at audit time:

- `https://notavello.com/pages/blog/ai-advice-needs-an-i-dont-know-step/`
- `https://notavello.com/pages/blog/macos-tahoe-efficiency-versus-windows-breadth-2026/`

Affected files:

- `.github/workflows/generate-daily-post.yml`
- `.github/workflows/generate-sitemap.yml`
- `.github/scripts/generate-sitemap.js`
- `sitemap.xml`

Cause: the post workflow commits generated pages but does not generate/stage the sitemap. Its Actions-authenticated push cannot be relied upon to launch a second workflow. The checked-in sitemap has 140 URLs and is missing the two newest indexable article URLs.

Recommendation: generate and validate `sitemap.xml` inside the same blog publication job before committing. Keep the standalone sitemap workflow for human-authored pushes.

Risk: simultaneous workflows can still race on `main`; use the existing concurrency group or make the sitemap check deterministic and rerunnable.

### 2. Sitemap generation does not enforce indexability

Affected file: `.github/scripts/generate-sitemap.js`.

The generator includes every eligible `index.html` based on path alone. It does not verify a self-canonical, reject `noindex`, reject canonical duplicates, or validate internal page metadata. It also cannot detect non-200 production responses.

Recommendation: make generation parse every candidate, require exactly one matching canonical, exclude `noindex`, reject query-string/noncanonical routes, and fail validation on metadata conflicts. Add an optional live crawl mode for production status/redirect checks.

Risk: strict validation will expose pre-existing non-blog pages with incomplete SEO metadata. Exclusions should be explicit and narrowly documented rather than silently accepted.

### 3. Sitemap has no meaningful `lastmod`

Affected files: `.github/scripts/generate-sitemap.js` and `sitemap.xml`.

No URL currently has a `lastmod`. This is not itself an indexing blocker, but it removes a useful recrawl signal. The existing generator comment intentionally avoids content-edit updates, which also prevents meaningful update dates.

Recommendation: use committed page metadata (`article:modified_time`, then `article:published_time`) for articles and Git's last content commit date for other pages. Dates must remain stable when content does not change.

Risk: filesystem modification times or build time would fabricate freshness and must not be used.

### 4. Shared navigation lacks a Blog link

Affected files: `_header.html` and `_footer.html`.

The homepage links to the blog directly, so the blog is already one click from `/`. However, the common header/footer injected into the rest of the site has no Blog link. This weakens site-wide discovery and topical connection.

Recommendation: add a normal `/pages/blog/` link to the shared footer (the smallest design-preserving change).

Risk: minimal layout wrapping on narrow screens; the existing footer already supports wrapping.

### 5. No automated crawlability gate

Affected files: `package.json` and `.github/workflows/`.

There are no working project tests (`npm test` intentionally exits 1) and no CI check for sitemap eligibility, noindex, broken links, canonical mismatch, robots blocking, or orphan articles.

Recommendation: add a dependency-free Node audit/check script, deterministic reports, an actual `npm test`, and a pull-request CI workflow.

Risk: external links and production availability are inherently flaky. CI should enforce repository-local links deterministically; live checks should be explicit rather than required for every PR.

### 6. Generator instructions appended to older article documents

Twenty-three older article files contained template/workflow instructions after the first closing `</html>` tag. The text was not part of the visible article body, but it was present in raw crawler responses and was a strong programmatic-content signal. The implementation removes only bytes after the closing document tag and adds a regression check; visible article content is unchanged.

## Confirmed healthy behavior

- Root `robots.txt` contains `User-agent: *`, `Allow: /`, and the correct sitemap URL.
- Current article files contain visible titles, descriptions, `index, follow`, and self-referencing HTTPS canonicals.
- Article main content is in each committed `index.html`, not fetched or assembled by client-side JavaScript.
- The Cloudflare Worker injects shared fragments before returning HTML; it does not branch on user agent and contains no crawler cloaking.
- A sampled live Googlebot request to the newest article returned HTTP 200 and the full head/article document. The live blog index also returned HTTP 200.
- `pages/blog/all-posts/index.html` contains ordinary `<a href="...">` links to all 95 current article directories found in the repository.
- The homepage contains a normal link to `/pages/blog/`, and paginated blog index pages provide archive traversal.
- No article-content rewrite is required or authorized by this audit.

## Suspected or externally dependent problems

### Crawl prioritization / content selection

The Search Console state with `Last crawled: N/A` can coexist with valid crawlability when Google has discovered more URLs than it currently chooses to fetch. The site publishes articles frequently, while many pages use a similar visual/editorial template and cover a broad topic range. This may reduce crawl priority or index selection, but Search Console exports, URL Inspection results, access logs, and editorial review are required to confirm it.

### Cloudflare security configuration

No repository code challenges crawlers. `_worker.js` can return 401/429/5xx only for the private `POST /api/generate-blog` endpoint; public GET pages pass through `env.ASSETS`. Account-level Cloudflare WAF, Bot Fight Mode, rate limiting, and access policies are not stored in this repository. Sampled live requests did not encounter them, but Cloudflare Security Events and response logs should be reviewed for Googlebot/Bingbot 403, 429, or challenge responses.

### URL normalization outside declared redirects

The intended form is HTTPS, lowercase, and trailing slash for directory pages. `_redirects` handles selected legacy routes, while host/scheme normalization and Cloudflare's directory behavior are deployment-level. A live variant matrix should be checked after implementation for HTTP, `www`, missing slash, uppercase paths, `.html`, and query strings. Query strings generally cannot be globally redirected without risking functional tool URLs; canonical tags and sitemap/internal-link hygiene are the safe baseline.

## Recommended implementation

1. Add a dependency-free SEO crawler/checker that builds a local URL graph, parses robots/sitemap/canonical/meta data, identifies duplicates/similarity/thin pages/orphans, and optionally verifies live HTTP behavior with both normal and Googlebot user agents.
2. Make the sitemap generator consume the same eligibility rules and stable content dates.
3. Run sitemap generation inside the daily-post workflow and stage the result with the post.
4. Add the Blog link to the shared footer.
5. Add CI and package scripts for deterministic checks and artifact reports.
6. Regenerate the archive/sitemap, run all checks, then use the live mode only as a diagnostic because production will not contain the branch changes until approved and deployed.

## Audit limitations

- Search Console data and its exact 51-URL export were not available in the repository.
- Cloudflare account settings/logs were not available.
- Only sampled live URLs were tested before implementation; the new live crawler will provide the repeatable full-site check.
- Google indexing is not guaranteed by technical compliance. The goal of these changes is reliable crawlability and unambiguous discovery signals.
